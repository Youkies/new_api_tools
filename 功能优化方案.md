# NewAPI Tools 功能优化方案

更新时间：2026-05-17

## 1. 文档目标

本文档用于承接当前 NewAPI Tools 的功能评估结论，把后续优化拆成可执行的产品、后端、前端和数据层任务。重点覆盖三个高价值页面：

- 仪表盘：从“统计展示页”升级为“运营驾驶舱”。
- 风控中心：从“排行 + IP 监控”升级为“风险队列 + 证据化处置中心”。
- 日志分析：从“即时查询 + 导出”升级为“可追踪、可下钻、可复用的分析平台”。

当前项目同时存在 Go 后端 `backend/` 与 Python 兼容后端 `backend-py/`。涉及接口契约变化时，应优先实现 Go 后端，并同步 Python 兼容接口，避免部署形态切换时功能语义不一致。

## 2. 总体优化原则

### 2.1 管理动作必须可解释、可预览、可回滚

凡是封禁、注销、移动分组、删除、清理等会影响用户状态的动作，都应满足：

- 执行前支持 `dry_run` 预览。
- 响应里返回命中条件、影响数量、样本用户和风险提示。
- 正式执行写入 `batch_id`、操作者、时间、条件快照、用户快照。
- 支持按批次查看结果，必要时支持撤销或补偿。

### 2.2 所有统计都要展示数据新鲜度

仪表盘、风控、日志分析、模型监控等页面都依赖大表统计。后续所有统计接口建议统一返回：

```json
{
  "snapshot_time": 1710000000,
  "cache_hit": true,
  "data_freshness": {
    "logs_max_created_at": 1710000000,
    "lag_seconds": 12,
    "source": "cache | live_query | rollup"
  }
}
```

前端应明确展示“数据更新时间”和“数据来源”，避免管理员把旧缓存误认为实时结果。

### 2.3 优先用聚合表和快照接口降低大表压力

`logs` 表是核心高压表。后续优化应尽量避免多个页面反复对 `logs` 做大窗口聚合：

- 高频统计使用小时/天级 rollup 表。
- 多模块同时需要的数据由后端 snapshot 接口一次聚合返回。
- 前端刷新应有规模感知，复用现有 `/api/system/scale`、`/api/system/warmup-status`、`refresh-estimate` 的思路。

### 2.4 页面之间要能下钻联动

管理工具的价值不只是“看数”，而是快速定位和处理问题。建议建立统一跳转参数：

- 仪表盘卡片点击进入日志分析，并带入时间、模型、渠道、用户等筛选。
- 风控风险用户点击进入用户详情，展示调用、IP、令牌、充值、封禁历史。
- 日志分析中的异常模型/渠道点击进入模型监控或成本核算。

## 4. 仪表盘优化方案

### 4.1 当前状态

前端 `frontend/src/components/Dashboard.tsx` 当前会分别请求：

- `/api/dashboard/overview`
- `/api/dashboard/usage`
- `/api/dashboard/models`
- `/api/dashboard/trends/hourly` 或 `/api/dashboard/trends/daily`
- `/api/dashboard/top-users`
- `/api/dashboard/system-info`
- `/api/dashboard/refresh-estimate`

Go 后端 `backend/internal/service/dashboard.go` 已有缓存、系统规模估算和部分 SQL 优化，但前端一次刷新仍会发出多组请求，不同卡片可能来自不同缓存时刻。

### 4.2 产品目标

仪表盘应变成“我现在要不要处理问题”的第一屏，而不是单纯展示静态指标。

建议第一屏保留：

- 今日请求数、今日消费、活跃用户、失败率。
- 与上一周期对比：例如较昨日同时间段 `+12.5%`。
- 数据新鲜度：最后日志时间、缓存时间、刷新建议。
- 异常入口：高失败率模型、异常渠道、风险用户、成本异常。

### 4.3 后端接口优化

新增聚合接口：

```http
GET /api/dashboard/snapshot?period=24h&trend_days=7&top_limit=10&no_cache=false
```

建议响应：

```json
{
  "period": "24h",
  "snapshot_time": 1710000000,
  "cache_hit": true,
  "refresh_hint": {
    "recommended_interval_seconds": 120,
    "reason": "large_logs_table"
  },
  "overview": {},
  "usage": {},
  "models": [],
  "trends": [],
  "top_users": [],
  "alerts": [],
  "data_freshness": {}
}
```

收益：

- 前端从 5-7 次请求降到 1 次主请求。
- 所有卡片共享同一 `snapshot_time`。
- 后端可统一控制缓存、降级和慢查询保护。

### 4.4 数据层优化

新增 rollup 表，避免频繁扫 `logs`：

```sql
api_tools_usage_rollup_hourly(
  bucket_hour,
  user_id,
  model_name,
  channel_id,
  request_count,
  success_count,
  error_count,
  prompt_tokens,
  completion_tokens,
  quota_sum,
  avg_use_time
)
```

```sql
api_tools_usage_rollup_daily(
  bucket_date,
  user_id,
  model_name,
  channel_id,
  request_count,
  success_count,
  error_count,
  prompt_tokens,
  completion_tokens,
  quota_sum,
  avg_use_time
)
```

Rollup 更新策略：

- 首期可用定时任务每 1-5 分钟增量聚合最近窗口。
- 保留 `last_processed_log_id` 或 `last_processed_created_at`。
- 支持手动刷新，后台防止重复运行。

### 4.5 前端体验优化

- 卡片支持点击下钻：
  - 请求数 -> 日志分析，带入时间范围。
  - 失败率 -> 日志分析，筛选错误类型。
  - Top 模型 -> 模型监控或日志分析。
  - Top 用户 -> 风控用户详情。
- 增加“异常提醒”区域：
  - 模型失败率突增。
  - 某渠道成本异常或毛利倒挂。
  - 某用户短时间请求暴涨。
- 刷新按钮区分：
  - 普通刷新：走缓存。
  - 强制刷新：带 `no_cache=true`，并展示预计耗时。

### 4.6 验收标准

- 仪表盘首屏主数据只依赖一个 snapshot 主接口。
- 所有关键卡片展示同一个 `snapshot_time`。
- 大日志表场景下，普通刷新不应触发多条长耗时 SQL。
- 卡片点击能正确跳到对应分析页并带入筛选参数。

## 5. 风控中心优化方案

### 5.1 当前状态

风控中心主要集中在 `frontend/src/components/RealtimeRanking.tsx`，同一个组件承载：

- 多窗口排行榜。
- 封禁列表。
- IP 监控。
- 审计日志。
- AI 封禁配置、扫描、待复核、白名单。
- 批量封禁和撤销逻辑。

后端已有：

- `backend/internal/service/risk_monitoring.go`：风险排行和用户风险分析。
- `backend/internal/service/ai_auto_ban.go`：AI 辅助封禁、待复核、审计日志、白名单。
- IP 监控相关接口：共享 IP、多 IP 用户、多 IP 令牌等。

当前最大问题不是“功能少”，而是“功能已经多到需要重新组织”。继续堆在一个页面里，会导致维护、测试、误操作控制都越来越困难。

### 5.2 信息架构重组

建议风控中心拆成 5 个页签：

1. 风险队列
   - 默认首页。
   - 按 `risk_score` 排序展示当前最值得处理的用户。
   - 每条记录展示风险原因、证据摘要、建议动作。

2. IP 案件
   - 聚合“同 IP 多用户”“同用户多 IP”“同 Token 多 IP”“短时跨地区跳 IP”。
   - 以案件为单位处理，而不是散落在多个列表里。

3. 用户画像
   - 用户详情下钻页。
   - 展示调用趋势、模型分布、IP 轨迹、令牌使用、充值记录、封禁/解封记录。

4. 处置记录
   - 封禁、解封、批量封禁、AI 待复核处理都进入统一审计流。
   - 支持按 `batch_id`、操作者、动作类型、风险原因过滤。

5. 策略配置
   - 风险评分权重。
   - IP 白名单/黑名单。
   - 受保护角色和分组。
   - AI 复核策略。

### 5.3 前端拆分计划

将 `RealtimeRanking.tsx` 拆为以下组件：

```text
frontend/src/components/risk/
  RiskCenter.tsx
  RiskQueue.tsx
  RiskQueueFilters.tsx
  AltAccountCasesPanel.tsx
  AltAccountAIResultCard.tsx
  IPCaseList.tsx
  UserRiskProfile.tsx
  RiskActionHistory.tsx
  AIBanPanel.tsx
  AIPendingReviewList.tsx
  RiskPolicySettings.tsx
  BulkActionDialog.tsx
  hooks/
    useRiskApi.ts
    useRiskRefresh.ts
    useRiskFilters.ts
```

拆分原则：

- 第一阶段只移动代码，不改行为。
- API 调用集中到 `useRiskApi`。
- 自动刷新集中到 `useRiskRefresh`。
- 批量操作弹窗统一使用 `BulkActionDialog`，避免每个列表各写一套确认逻辑。

当前已将共享 IP 小号研判收敛进统一案件工作台：

- `AltAccountCasesPanel.tsx` 承接“小号风险案件工作台”，`shared_ip` 与 30d 轮换账号池、邀请链、Token 轮换共用同一套列表、AI 研判结果和处置动作。
- `risk/types.ts` 沉淀 IP 监控相关类型，后续 `IPCaseList`、hooks 和其他风险组件应复用这些类型。
- 后端共享 IP 用户详情增加成功充值次数、总调用数、已用额度等字段，前端可直接标注“未充值”线索，适合识别批量注册/开小号。
- 共享 IP 案件已支持完整 IP 展示、用户展开、单用户分析/封禁、`封禁全部`、`撤销上一次` 和案件级 `AI研判`；AI 结果显示在对应案件行内，避免和共享 IP 监控形成两套 UI 模板。

### 5.4 风险队列设计

新增统一接口：

```http
GET /api/risk/queue?window=24h&page=1&page_size=50&sort=risk_score&order=desc
```

建议响应：

```json
{
  "items": [
    {
      "user_id": 123,
      "username": "demo",
      "status": 1,
      "risk_score": 86,
      "risk_level": "high",
      "suggested_action": "review",
      "metrics": {
        "request_count": 1200,
        "failure_rate": 0.18,
        "unique_ips": 9,
        "unique_tokens": 4,
        "quota_used": 123456
      },
      "reasons": [
        {
          "code": "HIGH_RPM",
          "label": "短时间高频请求",
          "severity": "high",
          "evidence": "1h 内 1200 次请求，高于阈值 500",
          "suggestion": "检查是否异常脚本调用"
        }
      ]
    }
  ],
  "total": 100,
  "snapshot_time": 1710000000
}
```

风险信号建议：

- 高频请求：短窗口请求数、RPM、并发趋势。
- 高失败率：错误率、空响应、超时。
- IP 异常：共享 IP、多 IP、短时地区跳变。
- 令牌异常：同用户多令牌轮换、同令牌多 IP。
- 成本异常：高消费、异常模型、成本倒挂。
- 用户属性：未充值高消耗、新注册高频、邀请链集中。
- 历史处置：曾被封禁、曾被 AI 标记、管理员忽略过。

### 5.5 风险评分规则

首期不要直接引入复杂模型，先做可解释分数：

```text
risk_score =
  request_velocity_score * 0.25 +
  failure_score * 0.20 +
  ip_anomaly_score * 0.20 +
  token_anomaly_score * 0.10 +
  cost_anomaly_score * 0.15 +
  history_score * 0.10
```

评分必须输出 `reasons[]`，不能只给一个数字。管理员真正需要的是“为什么这个用户排在前面”。

### 5.6 IP 案件优化

当前 IP 监控一次拉多组列表，适合拆为案件视图：

```http
GET /api/risk/ip-cases?window=24h&type=shared_ip&page=1&page_size=20
```

案件类型：

- `shared_ip`：同一 IP 多用户。
- `multi_ip_user`：同一用户短时间多 IP。
- `multi_ip_token`：同一令牌短时间多 IP。
- `ip_burst`：同一 IP 短时间请求暴涨。
- `geo_jump`：短时间跨地区跳变。

每个案件应返回：

- 案件摘要。
- 涉及用户数、请求数、封禁数。
- 未封禁用户列表。
- 推荐操作：观察、批量加入复核、批量封禁。
- 证据：时间窗口、IP、用户、请求量、失败率。

已落地的共享 IP 专项能力：

```http
POST /api/ai-ban/assess-shared-ip
```

请求：

```json
{
  "ip": "1.2.3.4",
  "window": "24h",
  "base_url": "可选，仅用于一次性效果检验",
  "api_key": "可选，仅用于一次性效果检验",
  "model": "可选，仅用于一次性效果检验"
}
```

说明：

- 常规前端调用只传 `ip/window`，使用后台已保存的 AI 配置。
- `base_url/api_key/model` 是一次性 override，不持久化到本地配置、文档或记忆库，适合临时效果检验。
- 后端先通过 `/api/ip/shared-users` 同口径拿案件数据，再将 `case_stats` 和最多 40 个用户明细发送给 AI。
- 返回字段包含 `case`、`case_stats`、`assessment`、`model`、`usage`、`assessed_at`。
- `assessment.action` 统一归一为 `monitor/review/ban`；即使模型返回自定义动作，也会按 `pending_review_first` 策略降级到可控枚举。

### 5.8 AI 风控优化

当前策略应继续坚持“先进入待复核”，不要默认自动封禁。

AI 评估输出建议标准化：

```json
{
  "risk_score": 8,
  "confidence": 0.92,
  "action": "review",
  "reason": "短时间高频请求且 IP 高频切换",
  "evidence_summary": [
    "1h 内请求 1200 次",
    "使用 9 个不同 IP",
    "失败率 18%"
  ],
  "false_positive_risk": "medium",
  "questions_for_admin": [
    "该用户是否为已知企业客户？",
    "这些 IP 是否来自可信代理池？"
  ],
  "prompt_version": "risk-ai-v2"
}
```

前端待复核队列动作：

- 确认封禁。
- 忽略一次。
- 加入白名单。
- 标记为误报。
- 只禁用令牌。
- 移入观察名单。

后端应记录每次管理员反馈，用于后续调整启发式规则和 AI prompt。

共享 IP 小号场景新增专项 Prompt：`shared-ip-alt-account-v1`。

重点判断：

- 同一 IP 下多个用户是否均未充值。
- 用户是否都只有极低调用量或极少 token，像批量试探账号。
- 首次出现时间是否高度集中。
- 是否混入已封禁账号或未封禁账号仍可继续调用。
- 共享 IP 是否可能是校园网、公司 NAT、家庭宽带、运营商 CGNAT、代理或 Cloudflare 出口，必须输出误报风险。

输出归一化要求：

- `risk_score` 归一到 `1-10`；模型若返回 `92` 这类百分制，后端会钳制到 10。
- `action` 只允许 `monitor/review/ban`；未知动作默认按风险分转为 `review` 或 `monitor`。
- `false_positive_risk` 只允许 `low/medium/high`；模型若返回对象，后端取 `level`，缺失则用 `medium`。
- `evidence_summary`、`questions_for_admin`、`likely_alt_account_users` 必须是数组；否则置为空数组，避免前端渲染异常。

2026-05-17 真实库 AI 效果检验：

| 项目 | 结果 |
| --- | --- |
| 数据来源 | 本地 `数据库地址.md` 指向的真实 MySQL，只读读取 24h 共享 IP 样本 |
| 模型入口 | `https://newapi.youkies.space/v1/chat/completions`，可见模型需使用 `「按量」gpt-5.5`，裸 `gpt-5.5` 在该 Key 默认分组下无可用通道 |
| 抽样 | 24h 内取 30 个共享 IP 样本，选中最高风险样本；管理员风控台和 AI 小号案件研判按当前口径使用完整 IP，不再脱敏 |
| 案件统计 | 15 个用户、15 个 token、18 次请求、15 个未封禁、15 个未充值、15 个低请求账号、首次出现跨度约 4171 秒 |
| AI 原始判断 | 高风险、小号/账号池倾向明显；建议先 `pending_review_first`，不直接永久封禁；输出 14 条证据、8 个复核问题、15 个疑似账号 |
| 接口加固 | 检验发现模型会返回非标准 `risk_score/action/false_positive_risk`，已在 Go/Python 后端增加归一化，保证前端拿到稳定 schema |

### 5.9 验收标准

- 风控中心主文件拆分后，单个组件不再承载所有业务。
- 风险列表默认按 `risk_score` 排序，且每条记录有 `reasons[]`。
- 批量封禁支持服务端 `dry_run`、正式执行和批次审计。
- AI 待复核记录可追踪 prompt 版本、证据、管理员反馈。
- 所有危险动作默认排除管理员和白名单用户。

## 6. 日志分析优化方案

### 6.1 当前状态

前端 `frontend/src/components/Analytics.tsx` 已支持：

- `/api/analytics/summary`：统计摘要。
- `/api/analytics/process`：触发日志处理。
- `/api/analytics/export`：按时间、类型、模型、用户、令牌、渠道、分组、Request ID 导出 CSV/JSON。

Go 后端 `backend/internal/service/log_analytics.go` 中，`ProcessLogs()` 当前语义更接近“清理缓存并重新统计”，并不是真正的后台同步处理。因此前端文案容易误导管理员，以为系统正在把日志从外部同步到本地。

### 6.2 P0 文案和语义修正

把前端“处理日志 / 同步日志”调整为：

- “刷新统计缓存”
- “重新计算日志摘要”
- “从当前数据库读取最新日志”

同时在响应里返回：

```json
{
  "processed": 0,
  "total_logs": 123456,
  "action": "cache_refresh",
  "message": "已刷新统计缓存，Go 后端直接从 logs 表读取数据"
}
```

这样可以避免管理员误解为存在后台同步队列。

### 6.3 分析能力增强

日志分析页建议增加 4 类分析视图：

1. 趋势视图
   - 请求量、消费、失败率、平均耗时。
   - 支持按模型、渠道、分组、用户切分。

2. 排行视图
   - 用户请求排行、消费排行。
   - 模型请求排行、成本排行。
   - 渠道请求排行、失败率排行。

3. 异常视图
   - 失败率突然升高。
   - 某模型平均耗时异常。
   - 某渠道成本或 quota 异常。
   - 某用户请求突增。

4. 明细视图
   - 保留现有导出能力。
   - 支持按 Request ID 快速定位。
   - 支持保存常用筛选条件。

### 6.4 导出任务化

当前 `/api/analytics/export` 适合中小范围导出。大范围导出建议任务化：

```http
POST /api/analytics/export-jobs
GET /api/analytics/export-jobs/{job_id}
GET /api/analytics/export-jobs/{job_id}/download
DELETE /api/analytics/export-jobs/{job_id}
```

导出任务状态：

```json
{
  "job_id": "export-20260517-0001",
  "status": "running",
  "progress": 0.42,
  "rows_written": 42000,
  "estimated_total_rows": 100000,
  "file_size": 12345678,
  "created_at": 1710000000,
  "expires_at": 1710086400
}
```

收益：

- 大导出不中断页面。
- 可显示进度、取消、重试。
- 文件可短期保留，避免重复查询。

### 6.5 与其他模块联动

- 仪表盘点击卡片进入日志分析。
- 风控用户详情里的“查看原始日志”进入日志分析并带入用户和时间窗口。
- 成本核算可跳到日志分析查看毛利异常的原始记录。
- 日志对账可跳回日志分析查看未匹配的本站日志。

### 6.6 验收标准

- Go 后端的“处理日志”文案不再暗示外部同步。
- 导出大范围日志时，前端能看到进度或明确提示。
- 日志分析支持保存筛选条件。
- 异常视图至少覆盖失败率、耗时、消费突增三类。

## 7. 其他模块优化建议

### 7.1 用户管理

当前已将批量注销口径收紧为“未成功充值且从未调用过”。后续建议：

- 所有批量动作统一走 `dry_run -> confirm -> execute`。
- 用户列表支持保存筛选条件。
- 用户详情整合充值、调用、令牌、分组迁移、封禁历史。
- 危险操作统一显示命中条件和排除条件，例如是否排除管理员、白名单、已充值用户。

### 7.2 自动分组

建议继续强化“可解释迁移”：

- 扫描结果显示每个用户命中的规则。
- 手动迁移和自动迁移共用批次记录。
- 自动分组日志支持按 `batch_id` 回放和撤销。
- 增加“不会降级 / 不移动自定义分组”的显式说明。

### 7.3 成本核算

建议从“成本汇总”升级为“毛利监控”：

- 展示渠道、模型、分组、用户维度毛利。
- 标记毛利为负或异常偏低的组合。
- 与模型监控联动：模型不可用、失败率高、成本异常同屏提示。
- 支持成本规则变更前后的影响预览。

### 7.4 日志对账

当前 upload-only key 设计是正确方向。后续建议：

- 上传文件增加来源站点、时间范围、hash 去重。
- 分析结果支持保存历史报告。
- 未匹配项支持一键导出为排查 CSV。
- 渠道 alias 和模型标准化规则做成可视化配置。

### 7.5 模型监控

建议补充：

- 模型状态历史趋势。
- 渠道级故障聚合。
- 与日志失败率联动，区分“探测失败”和“真实用户请求失败”。
- 支持按模型快速跳转到日志分析。

### 7.6 充值记录和兑换码

建议重点补审计能力：

- 充值记录增加异常检测：重复到账、支付成功但未到账、过期仍待支付。
- 兑换码批量生成增加规则摘要和二次确认。
- 兑换码删除/清理写入批次审计。

### 7.7 运营预警

已新增独立 `运营预警` 页面，定位为“保收入、保体验、早发现事故”，不并入自动封禁。

v1 已覆盖：

- 高价值用户突然停用：历史高消耗或高请求用户 72h 无调用。
- 高充值用户充值断档：按历史成功充值间隔判断是否明显超期。
- 新充值用户未激活：充值后无调用，或充值后失败率偏高。
- 付费用户体验异常：高价值用户 24h 失败率或平均响应耗时异常。
- 支付状态异常：15 分钟以上待处理订单、疑似重复成功到账订单。

明确边界：

- 收入/毛利异常暂不做，因为当前没有接入上游真实价格系统，避免输出不可靠的成本/利润判断。
- 运营预警页点开用户详情时直接展示注册邮箱，便于管理员人工联系。
- 注册邮箱不发送给外部 AI，不作为默认导出字段扩散。

后续建议：

- 将“已处理/忽略/观察中”从前端本地状态升级为服务端持久化。
- 增加运营预警配置页，允许调整高价值阈值、静默时间、充值断档阈值和支付超时阈值。
- 与用户管理详情、日志分析筛选、充值记录详情建立更完整的下钻参数。

目标：让风控中心具备“先看谁最危险”的默认工作流。

- 后端新增 `/api/risk/queue`。

- 新增 dry-run、execute、revert 接口。

## 9. 推荐的第一批任务

## 10. 长期验收口径

优化完成后，NewAPI Tools 应满足：

- 管理员打开仪表盘，能在 10 秒内知道系统是否异常。
- 管理员打开风控中心，默认看到的是按证据排序的风险队列，而不是散落的多个排行榜。
- 每个封禁/注销/迁移动作都能回答“谁在什么时候因为什么条件影响了哪些用户”。
- 日志分析不只是导出工具，而是仪表盘、风控、成本、日志对账的共同下钻入口。
- 大规模日志表下，普通刷新不依赖多次全表聚合。
- AI 风控默认辅助复核，除非显式开启高置信自动执行。
